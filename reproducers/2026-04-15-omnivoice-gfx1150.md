# In-the-Wild Reproducer: OmniVoice voice cloning on gfx1150

**Date observed:** 2026-04-14 / 2026-04-15
**Bug exercised:** [#3 — PyTorch workspace=0 to MIOpen Gemm solvers](../README.md#bug-3-pytorch-workspace-allocation-upstream-pytorch-issue)
**Severity:** Rate-limits production workload by ~3.5×

## Workload

- **Model:** [k2-fsa/OmniVoice](https://github.com/k2-fsa/OmniVoice) — diffusion language model for zero-shot text-to-speech voice cloning
- **Architecture:** transformer-based diffusion TTS with conv-heavy denoising backbone
- **Input:** 6-second voice reference WAV + text transcript → ~20 seconds of synthesized cloned voice audio
- **Typical call:** model loaded on GPU (`device_map="cuda:0"`, `dtype=torch.float16`), forward pass through ~24 diffusion steps per second of output

## Hardware / stack

- GPD Pocket 4, AMD Ryzen AI 9 HX 370 (Strix Point), Radeon 890M iGPU (gfx1150)
- 32 GB unified memory
- CachyOS, kernel `6.19.12-1-cachyos`
- ROCm 7.2.0, MIOpen from matching stack (patched with `0001-add-gfx1150-to-ck-whitelists.patch` from this repo's `patches/`)
- Custom-built PyTorch `2.12.0a0+gitf86ea9e`, HIP `7.2.26043`, targeting `gfx1150`, built against Python 3.14
- No CK illegal-opcode crash observed (bug #2 not triggered by this workload)
- No hipBLAS Tensile failure observed

## Performance impact (measured)

Same generation (6-second reference audio, ~80-word input text, ~20 seconds of output audio), identical code path, identical input, averaged over multiple runs:

| Configuration | Wall time | Speedup vs baseline |
|---|---|---|
| **Baseline** (no env workaround) | **2m 52s** | 1.0× |
| `MIOPEN_FIND_MODE=FAST`, cold cache / cold ROCm driver | 1m 20s | 2.15× |
| `MIOPEN_FIND_MODE=FAST`, warm cache / warm ROCm driver | **0m 49s** | **3.51×** |

**The baseline is rate-limited by the workspace fallback.** Every `FindSolutions` + `EvaluateInvokers` double-call across 40+ conv shapes in a single forward pass adds up. With `MIOPEN_FIND_MODE=FAST` the find DB is consulted directly and the fallback path is mostly avoided. Quality of the generated audio is **indistinguishable** between baseline and the FAST-flagged runs (verified by A/B listening).

**Output quality**: unchanged across configurations. The fallback is slow but functionally correct.

## Why this reproducer matters

1. **It's a production-shaped workload**, not a microbenchmark. Diffusion TTS with reference-conditioned generation is a normal user-facing use case. The bug isn't theoretical — it directly costs the user 2+ minutes per voice clone on this hardware.

2. **It's the second distinct production reproducer on gfx1150/gfx1151 class hardware**. The first is [TimLawrenz's NanoDiT training run on gfx1151 Strix Halo](https://github.com/ROCm/TheRock/issues/2591) — a 400M parameter Diffusion Transformer for text-to-image at batch 16, bf16, 1024px resolution. Both workloads are conv-heavy, both exercise the same `GemmFwdRest` fallback, both hit the same kernel family in hipBLASLt's Cijk path. That's two independent applications on two adjacent chips (Strix Point and Strix Halo) with the same rate limiter.

3. **It provides concrete workspace sizes** that MIOpen is failing to satisfy (up to 424 MB). Any upstream fix proposal needs to handle workspaces in that range.

4. **It confirms the community workaround** (`MIOPEN_FIND_MODE=FAST`) works on gfx1150, not just gfx1151. Previously documented only for [Chatterbox TTS on Strix Halo by bkpaine](https://medium.com/@bkpaine1/voice-cloning-on-amd-strix-halo-running-chatterbox-tts-with-native-gpu-acceleration-fa4a3db5e82c). This reproducer extends the coverage.

5. **The `FAST` workaround isn't a full fix** — it routes around the bug, it doesn't fix the underlying workspace=0 passthrough from PyTorch. If the PyTorch side were patched to pass real workspace budgets (as MIOpen's Find2 API expects), we would expect further speedup beyond the current 3.5×, likely into the 5-8× range based on similar conv workloads on healthy stacks.

## Reproduction

Full step-by-step at [Peterc3-dev/omnivoice-gfx1150](https://github.com/Peterc3-dev/omnivoice-gfx1150) — see `REPRODUCTION-GUIDE.md` and `bench.py` in that repo.

Short version:

1. Build PyTorch `2.12.0a0+gitf86ea9e` (or newer) targeting `gfx1150` against Python 3.14 — see [Peterc3-dev/pytorch-gfx1150](https://github.com/Peterc3-dev/pytorch-gfx1150)
2. Create Python 3.14 venv, expose the torch build via `.pth` file
3. Install OmniVoice with the torchaudio workaround (`torchaudio==2.9.1` with `_IS_TORCHAUDIO_EXT_AVAILABLE=False` patch — OmniVoice only uses pure-Python torchaudio ops)
4. Run `omnivoice-infer` with `--device cuda`, observe `MIOpen(HIP): Warning [IsEnoughWorkspace] Solver <GemmFwdRest>` in stderr
5. Repeat with `MIOPEN_FIND_MODE=FAST` env var set, measure wall time delta

## Interaction with this repo's patches

- **Bug #1 patch (CK whitelist)** — applied and in use. Doesn't affect this reproducer because OmniVoice doesn't hit CK-capable conv shapes.
- **Bug #2 (CK device code crash)** — not triggered by this workload. No illegal-opcode failures observed.
- **Bug #3 (workspace=0)** — **this is the reproducer**. Not yet patched upstream. `MIOPEN_FIND_MODE=FAST` is the current workaround.
